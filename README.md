# Searay Sales Intelligence

Proof-of-concept sales intelligence tool for jewellery wholesale reps. It ingests Unleashed ERP exports, scores customers with RFM analysis, matches upsell opportunities from the product catalogue, and surfaces three actionable recommendations per visit — via one-page Markdown reports and a mobile-first web app with a rule-based learning loop.

**Target customers (POC):** Class A Manufacturing Jewellers, Atlas Jewellers, My Jewellers, The Cut Jewellery, and Noonan's Showcase Jewellers.

## Features

- **RFM scoring** — Recency, frequency, and monetary bands with segment assignment and relationship flags from meeting notes
- **Upsell engine** — New-since-last-order products inside bought groups, backorder split, and white-space group discovery
- **One-page sales reports** — Markdown action reports with real product codes, prices, and head-office approval flags
- **Rep-facing web app** — Mobile-first UI to review actions, accept/reject with reason chips, and get better suggestions
- **Learning loop** — Rejection reasons (price ceiling, group exclusion, no discounts, low pressure) persist per customer and feed back into reports

## Requirements

- Python 3.11+
- [uv](https://docs.astral.sh/uv/) for environment management
- Unleashed export CSVs (see [Data setup](#data-setup))

No third-party Python packages — the project uses the standard library only.

## Quick start

```bash
# Clone and enter the project
cd SalesIntelligentApp

# Install uv if needed: https://docs.astral.sh/uv/getting-started/installation/

# Sync the virtual environment (creates .venv/)
uv sync

# Place Unleashed exports in Example Data/ (see below), then:

# Print RFM scorecard + upsell matching (checkpoint, no files written)
uv run python analyze.py

# Generate Markdown reports for all five customers
uv run python build_reports.py

# Start the rep web app
uv run python server.py
```

Open **http://localhost:8000** on your computer, or the LAN URL printed in the terminal on a phone on the same Wi-Fi.

## Scripts

| Command | Purpose |
|---------|---------|
| `uv run python analyze.py` | Step 2+3 checkpoint — prints RFM scorecard and per-customer product matching to the console |
| `uv run python build_reports.py` | Step 4 — writes `reports/<code> - <name>.md` for each target customer |
| `uv run python server.py` | Serves the rep web app and JSON API on port 8000 (override with `PORT` env var) |

Generated output:

- `reports/` — Markdown sales action reports (regenerated by `build_reports.py`)
- `feedback/` — Per-customer learned preferences from the web app (runtime state)

## Web app

The rep GUI (`webapp/`) is served by a zero-dependency stdlib HTTP server.

1. Pick a customer from the list
2. Review three recommended actions
3. Tap **Good** or **No** on each action
4. On rejection, select reason chips (price, group, discounts, pressure, etc.)
5. Tap **Show me 3 better** to get a refreshed set informed by what was learned

Deep-link to a customer: `http://localhost:8000/#c=MJ001`

### API

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/reasons` | Rejection reason chips |
| `GET` | `/api/customers` | Customer list with snapshot metadata |
| `GET` | `/api/customer/<code>` | Customer detail and current three actions |
| `POST` | `/api/customer/<code>/feedback` | Body: `{ "accepted": [id], "rejections": [{ "id", "reasons", "note" }] }` |
| `POST` | `/api/customer/<code>/reset` | Clear learned preferences for that customer |

### How recommendations work

1. **Profiles** — `utils/profile.py` builds RFM scorecards and purchase history from sales/invoice data
2. **Candidates** — `utils/candidates.py` tags actionable upsell and strategy items from the catalogue and meeting notes
3. **Preferences** — `utils/preferences.py` stores per-customer accept/reject learning in `feedback/<code>.json`
4. **Recommender** — `utils/recommend.py` is the single source of truth for the current top three actions (used by both the server and `build_reports.py`)

## Deployment (field use)

The web app is designed for reps on their phone during customer visits. No cloud hosting is required for the POC — run the server on a laptop on the same Wi-Fi network.

```bash
uv run python server.py
# or on a custom port:
PORT=8080 uv run python server.py
```

The server binds to `0.0.0.0`, so other devices on the LAN can reach it.

1. Start the server on your laptop
2. Find the laptop's LAN IP (e.g. **System Settings → Network** on macOS, or `ipconfig` / `ifconfig`)
3. On the phone, open `http://<laptop-ip>:8000`
4. Share a direct link to a customer: `http://<laptop-ip>:8000/#c=MJ001`

**Notes**

- Keep the laptop awake and on the same Wi-Fi as the phone
- macOS firewall may prompt to allow incoming connections the first time — allow Python
- Learned preferences are saved locally in `feedback/`; back up that folder if you want to preserve rep feedback between sessions
- Re-run `build_reports.py` after feedback sessions to refresh Markdown reports with the latest learned top-three actions

This POC is not production-hardened (no auth, HTTPS, or multi-user isolation). Treat it as a local demo tool only.

## Development

### Conventions

- **`constants/`** — Enums, config, and domain constants (RFM bands, rejection reasons, meeting notes). Avoid raw string/int comparisons in logic; use these types instead.
- **`utils/`** — Reusable functions (CSV I/O, scoring, recommendations, report rendering).
- **Entry points** — Top-level scripts (`analyze.py`, `build_reports.py`, `server.py`) orchestrate utils; keep business logic out of them where possible.
- **Dependencies** — Standard library only. Add new packages with `uv add <package>` if needed.
- **Changelog** — Behaviour changes are logged in [CHANGELOG.md](CHANGELOG.md).

### Typical workflow

```bash
# After changing RFM bands, meeting notes, or customer list:
uv run python analyze.py          # sanity-check scorecard + matching

# After changing report layout or recommendation logic:
uv run python build_reports.py    # regenerate reports/

# After changing the web app or API:
uv run python server.py           # manual test on desktop + phone
```

### Updating source data

When new Unleashed exports arrive:

1. Drop the four CSVs into `Example Data/` (update filenames in `constants/config.py` if the export date changes)
2. Update `ANCHOR_DATE` in `constants/config.py` to match the export snapshot
3. Re-run `analyze.py`, then `build_reports.py`
4. Delete `feedback/` if you want a clean learning slate for the new snapshot

### Gitignored paths

Do not commit customer/sales data or runtime state:

## Troubleshooting

| Problem | Fix |
|---------|-----|
| `FileNotFoundError` on CSV load | Confirm all four files are in `Example Data/` with exact names from `constants/config.py` |
| Phone cannot reach the app | Same Wi-Fi, correct LAN IP, firewall allows Python, server still running |
| `Address already in use` (port 8000) | Another `server.py` is still running — `lsof -nP -iTCP:8000 -sTCP:LISTEN` then `kill <PID>`, or use `PORT=8080 uv run python server.py` |
| Empty customer list or actions | Check that target customer codes exist in the sales/invoice exports |
| Reports differ from web app | Run `build_reports.py` again — both use `utils/recommend.py`, but reports are only refreshed on build |
| Garbled characters in console | Expected for latin-1 exports; file reads use the correct encoding per `FileSpec` |
